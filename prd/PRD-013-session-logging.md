# PRD-013: Session-Aware Query Logging

**Status:** Draft  
**Author:** metabot  
**Date:** 2026-04-25

---

## Problem

The current query log records the query string, total result count, and latency. The click log records which docno was clicked and at what rank. But the two logs are not linked — there is no way to reconstruct a user session. Given a click on rank 2, you cannot know what was at ranks 1, 3–10. Given two queries in sequence, you cannot tell if they came from the same user in the same session or from two different people.

This makes the logs nearly useless for ranking evaluation. To judge whether a click on rank 5 means rank 5 was surprisingly good or the top 4 results were just bad, you need the full result list. To build dwell-time or reformulation signals ("user searched, clicked rank 2, came back after 4 seconds, clicked rank 5 — rank 2 was probably not what they wanted"), you need session continuity.

---

## Goal

Enrich logging so that any user session can be fully replayed:

> User (IP 1.2.3.4) searched "einstein" at 13:00:00 and saw these 10 results.  
> At 13:00:04 they clicked rank 2 (Albert_Einstein).  
> At 13:00:08 they clicked rank 5 (Theory_of_relativity).  
> At 13:00:45 they searched "special relativity" in the same session.

---

## Design

### Session ID

A session is identified by a `session_id`: a random token generated by the browser on first page load and stored in `sessionStorage` (not `localStorage` — it should not persist across browser restarts). The frontend sends it with every search and click request. The server logs it but does not validate or track it — it is purely a correlation key for log analysis.

`sessionStorage` is the right scope: it survives page navigations within a tab but not new tabs or browser restarts. A new tab = a new session, which is the right granularity for search behaviour analysis.

The token is 16 hex characters (64 bits of randomness), generated once per session:

```javascript
if (!sessionStorage.getItem('zet_sid')) {
    sessionStorage.setItem('zet_sid', Math.random().toString(36).slice(2, 10) +
                                      Math.random().toString(36).slice(2, 10));
}
const SESSION_ID = sessionStorage.getItem('zet_sid');
```

### IP Address

The server logs the client IP from the request. Behind Caddy, the real IP is in the `X-Forwarded-For` header. Use the leftmost address (the actual client), not the proxy address.

IPs are logged as-is for now. No anonymisation, no geolocation. If privacy requirements change this is a one-line change.

### Query log changes

**Current:**
```json
{"ts": "2026-04-25T13:00:00Z", "q": "einstein", "total": 847, "took_ms": 18.3}
```

**New:**
```json
{
  "ts": "2026-04-25T13:00:00Z",
  "sid": "a3f9bc12de45678f",
  "ip": "1.2.3.4",
  "q": "einstein",
  "total": 847,
  "took_ms": 18.3,
  "results": [
    {"rank": 1, "docno": "Albert_Einstein",         "score": 9.82},
    {"rank": 2, "docno": "Theory_of_relativity",    "score": 7.41},
    {"rank": 3, "docno": "Special_relativity",      "score": 6.88},
    ...up to 20 results...
  ]
}
```

Log up to 20 results regardless of how many the user requested (`n` parameter). This is the "what did they see" record. If the query returned fewer than 20 results, log all of them.

### Click log changes

**Current:**
```json
{"ts": "2026-04-25T13:00:04Z", "q": "einstein", "docno": "Albert_Einstein", "rank": 2, "score": 9.82}
```

**New:**
```json
{
  "ts": "2026-04-25T13:00:04Z",
  "sid": "a3f9bc12de45678f",
  "ip": "1.2.3.4",
  "q": "einstein",
  "docno": "Albert_Einstein",
  "rank": 2,
  "score": 9.82
}
```

The click log does not need to repeat the full result list — that's already in the query log entry with the same `sid` + `q`. Join on `(sid, q)` to reconstruct the full context of any click.

### Crash log

No changes — crash log is operational, not analytical. IP and session are not relevant there.

## API Changes

### `/search` endpoint

Add `sid` as an optional query parameter. If absent, log as `null`.

```
GET /search?q=einstein&n=10&sid=a3f9bc12de45678f
```

Add IP extraction from `X-Forwarded-For` header, falling back to the direct connection IP.

### `/click` endpoint

Add `sid` and `ip` to the `ClickEvent` body. `sid` comes from the frontend. `ip` is extracted server-side (same as `/search`).

```json
{"q": "einstein", "docno": "Albert_Einstein", "rank": 2, "score": 9.82, "sid": "a3f9bc12de45678f"}
```

## Frontend Changes

**`index.html`** — two changes:

1. Generate and persist `SESSION_ID` in `sessionStorage` on page load.
2. Pass `sid` as a query parameter on every `/search` call and in every `/click` POST body.

This is ~10 lines of JavaScript.

## Session Replay

To reconstruct the example session from the problem statement:

```python
import json

sid = "a3f9bc12de45678f"
queries = [json.loads(l) for l in open("queries.jsonl") if f'"sid": "{sid}"' in l]
clicks  = [json.loads(l) for l in open("clicks.jsonl")  if f'"sid": "{sid}"' in l]

events = sorted(queries + clicks, key=lambda e: e["ts"])
for e in events:
    if "results" in e:
        print(f"{e['ts']}  SEARCH '{e['q']}' → {len(e['results'])} results shown")
        for r in e["results"][:5]:
            print(f"    {r['rank']}. {r['docno']}  ({r['score']})")
    else:
        print(f"{e['ts']}  CLICK  '{e['q']}' rank {e['rank']} → {e['docno']}")
```

Output:
```
2026-04-25T13:00:00Z  SEARCH 'einstein' → 10 results shown
    1. Albert_Einstein  (9.82)
    2. Theory_of_relativity  (7.41)
    3. Special_relativity  (6.88)
    4. General_relativity  (6.72)
    5. E=mc²  (6.11)
2026-04-25T13:00:04Z  CLICK  'einstein' rank 2 → Theory_of_relativity
2026-04-25T13:00:08Z  CLICK  'einstein' rank 5 → E=mc²
2026-04-25T13:00:45Z  SEARCH 'special relativity' → 10 results shown
    ...
```

## Log Volume

Currently the query log grows by one short line per query. With the results list added, each query log entry becomes roughly 20× larger (~1 KB vs ~50 bytes). At low query volumes (say 1,000 queries/day) this is ~1 MB/day — negligible. At high volumes this would need rotation, but that's not a near-term concern.

Log rotation (via `logrotate` or Python's `RotatingFileHandler`) is out of scope for this PRD but should be added before the logs grow past a few GB.

## What This Enables (Not in Scope Here)

- **Click-through rate by rank**: what fraction of queries get a click at rank 1 vs rank 2 vs rank 5?
- **Abandonment detection**: sessions that search but never click — are results just bad?
- **Reformulation detection**: same session, two queries in sequence — did the second query follow from a bad result on the first?
- **Click prior improvement**: actual click data from real queries can supplement the Wikipedia clickstream signal with site-specific popularity
- **MRR / nDCG estimation**: with assumed relevance (click = relevant), compute ranking quality metrics over time

---

## Success Criteria

1. Every search request logged with `sid`, `ip`, and up to 20 results
2. Every click logged with `sid` and `ip`
3. Given a `sid`, can reconstruct the full sequence of queries and clicks for that session with a simple log grep + sort
4. IP correctly reflects the real client address, not Caddy's loopback IP
5. No performance impact — logging is fire-and-forget (`asyncio.create_task`), unchanged from current behaviour
